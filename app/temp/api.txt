그럼 헷갈리니까 코드를 처음부터 다시 짜줘. 그리고 아래 내용을 모두 주석으로 설명도 추가해줘.

1. main.py : fast api앱 등록, lifespan 사용함(당장안에 내용은 없어도 생성), 이벤트 핸들러 등록, 라우터 등록
2. router.py : 엔드포인트 두개 등록, 첫번째: 모델 리스트 조회(근데 옵셔널로 모델의 상태값을 받음, 비즈니스 로직에서는 이 상태값을 같이 인퍼런스 서버로 전달해줌. 그럼 인퍼런스 서버가 알아서 상태값에 맞는 리스트만을 반환), 두번째: 모델을 인퍼런스 서버에 등록함. (이때 필수로 받는게 모델이름이랑 모델파일임) - 이걸 redis + celery 사용하도록함.
3. infra/ : celery_client.py, inference_client.py -> 인퍼런스 서버랑 통신, celery 클라이언트 정의
4. 스키마 : DTO 구조를 사용할것임, Request -> Response 구조 사용할 수 있도록 스키마 정의, exception 관련해서 이벤트 핸들러 사용할 수 있도록 exception 관련 스키마 정의할것
5. tasks/model_task.py : celery 사용을 위한 데코레이터 설정
6. services/model_service.py : 라우터에서 정의한 2개의 api의 실제 비즈니스 로직처리, 지금 인퍼런스 서버를 연결할 수없으니 우선 테스트 값으로 하되, 실제 구현때 사용할수 있는 코드를 주석으로 제공할것. 그리고 DTO 패턴?(구조?)를 사용하는 방식으로 구성할것
- api1 : 모델 리스트 조회  (동기방식) -> exception 이벤트 핸들러 사용. 인퍼런스 서버에서 돌려주는 에러코드를 기반으로 우리 서빙서버도 에러코드를 냄. 아니면 필수값이 비었다거나하도록... 하는거 암튼 고려해서 excepiton 핸들링할것. 
- API2. 모델 등록 접수(비동기 트리거), 실제 백그라운드 로직(Celery worker) 로 작업 할수있ㄷ록할것. 물론 DTO 유지되어야함. 그리고 이벤트핸들러 이용 + (celery + pooling) 식으로 작업할 것임 . 파일 업로드 중은 pooling을 할수있도록해서 exception 처리할 수 있도록할것.
